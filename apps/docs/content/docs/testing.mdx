Because `@onrails` workflows are composed of pure data and deterministic transformations, testing them is incredibly clean. Failures are represented as values (`Result<T, E>` / `Maybe<T>`) rather than side-effects (thrown exceptions). You do not need complex nested `try`/`catch` blocks, frame-level monkey-patching, or heavy mock frameworks to test your application logic.

This guide demonstrates key testing patterns for code using `@onrails` primitives and the **Railway-Oriented Programming (ROP)** model.

---

## 1. Direct Assertions & Pattern Matching

The most fundamental testing pattern is asserting on the outcome of a function returning a `Result` or `Maybe`.

### Testing Ok and Err branches
The type guards `isOk` and `isErr` narrow the TypeScript type inside the branch, ensuring that once you assert a branch, accessing the payload is fully type-safe.

```ts title="parse-id.spec.ts"
import { describe, expect, it } from "bun:test";
import { isErr, isOk, unwrapOk } from "@onrails/result";
import { parseId } from "../src/parse-id";

describe("parseId", () => {
  it("returns Ok for a numeric string", () => {
    const result = parseId("42");
    expect(isOk(result)).toBe(true);
    
    // Type is narrowed to Ok<number, string>, allowing type-safe extraction
    expect(unwrapOk(result)).toBe(42);
  });

  it("returns a typed Err for garbage", () => {
    const result = parseId("nope");
    expect(isErr(result)).toBe(true);
    expect(result.error).toBe("Not a number");
  });
});
```

### Resolving `ResultAsync`
`ResultAsync` is thenable. Resolving it (using `await` directly or via `.resolve()`) yields a standard `Result` that you can narrow and assert on.

```ts title="async-service.spec.ts"
import { expect, it } from "bun:test";
import { isOk, unwrapOk } from "@onrails/result";
import { fetchUser } from "../src/fetch-user";

it("resolves to a bare Result", async () => {
  // ResultAsync can be directly awaited
  const result = await fetchUser("usr_100");
  
  expect(isOk(result)).toBe(true);
  expect(unwrapOk(result).name).toBe("Ada");
});
```

### Collapsing paths with `match`
If your test cares only about the final mapped outcome rather than individual channel assertions, you can fold both the success and failure branches into a single value using `match`.

```ts title="respond.spec.ts"
import { expect, it } from "bun:test";
import { match } from "@onrails/result";
import { parseId } from "../src/parse-id";

const getResponseStatus = (input: string) =>
  match(
    parseId(input),
    () => 200,          // Ok track
    () => 400,          // Err track
  );

it("folds Ok and Err into a status code", () => {
  expect(getResponseStatus("42")).toBe(200);
  expect(getResponseStatus("nope")).toBe(400);
});
```

## 2. Testing Workflow Short-Circuiting

A core behavior of any railway pipeline is that it executes all steps on the happy path, but short-circuits on the very first error (the sad path). To test a workflow, you should verify both paths by feeding inputs that succeed or fail at specific boundaries.

### Example Workflow
Consider a `checkoutWorkflow` built with the fluent [Railway](file:///Users/alanrsoares/dev/onrails/.agents/skills/railway-do-notation/SKILL.md#layer-2--fluent-railway) builder:

```ts title="checkout.ts"
import { Railway } from "@onrails/result/railway";
import { type ResultAsync, ok, err } from "@onrails/result";

export type Cart = { items: Array<{ sku: string; qty: number }> };
export type Inventory = { sku: string; stock: number };
export type Order = { id: string; total: number };

export interface CheckoutDeps {
  findStock(sku: string): Promise<Inventory | null>;
  chargeCard(cart: Cart): Promise<Order>;
}

export const checkoutWorkflow = (
  deps: CheckoutDeps,
  cart: Cart
): ResultAsync<Order, Error> => {
  const item = cart.items[0];
  if (!item) {
    return ResultAsync.err(new Error("CartIsEmpty"));
  }

  return Railway.context({ cart, item })
    .fromPromise(
      "stockRow",
      ({ item }) => deps.findStock(item.sku),
      (e) => new Error(`Stock check failed: ${String(e)}`)
    )
    .require(
      "inventory",
      "stockRow",
      () => new Error("ItemNotFound")
    )
    .fromResult(
      "inStock",
      ({ item, inventory }) =>
        inventory.stock >= item.qty
          ? ok(inventory)
          : err(new Error("OutOfStock"))
    )
    .fromPromise(
      "order",
      ({ cart }) => deps.chargeCard(cart),
      (e) => new Error(`Payment failed: ${String(e)}`)
    )
    .select(({ order }) => order);
};
```

### Testing Happy & Sad Paths
To test this, pass simple mock dependencies (using dependency injection) and verify that the error track is hit when stock is insufficient, skipping the payment step entirely.

```ts title="checkout.spec.ts"
import { describe, expect, it } from "bun:test";
import { isErr, isOk } from "@onrails/result";
import { checkoutWorkflow, type CheckoutDeps } from "./checkout";

describe("checkoutWorkflow", () => {
  it("executes fully on the happy path", async () => {
    let paymentCalled = false;
    const mockDeps: CheckoutDeps = {
      findStock: async (sku) => ({ sku, stock: 10 }),
      chargeCard: async (cart) => {
        paymentCalled = true;
        return { id: "ord_123", total: 4200 };
      }
    };

    const result = await checkoutWorkflow(mockDeps, {
      items: [{ sku: "item-1", qty: 2 }]
    });

    expect(isOk(result)).toBe(true);
    expect(paymentCalled).toBe(true);
  });

  it("short-circuits and skips payment when inventory is insufficient", async () => {
    let paymentCalled = false;
    const mockDeps: CheckoutDeps = {
      // Stock is less than the requested quantity (2)
      findStock: async (sku) => ({ sku, stock: 1 }),
      chargeCard: async (cart) => {
        paymentCalled = true; // should never be reached
        return { id: "ord_123", total: 4200 };
      }
    };

    const result = await checkoutWorkflow(mockDeps, {
      items: [{ sku: "item-1", qty: 2 }]
    });

    expect(isErr(result)).toBe(true);
    expect(result.error.message).toBe("OutOfStock");
    expect(paymentCalled).toBe(false); // verification of short-circuit
  });
});
```

## 3. Testing Workflow Steps in Isolation

`Railway` steps are just functions of the accumulated context that return a value, `Result`, or `ResultAsync`. Declare them as standalone functions instead of inline lambdas and **each step can be exported and tested in isolation** without running the entire workflow or stubbing other async steps.

### Defining Reusable Steps
```ts title="profile-steps.ts"
import { trySync, tryAsync, type Result, type ResultAsync } from "@onrails/result";
import { z } from "zod";

const toError = (e: unknown) => e instanceof Error ? e : new Error(String(e));

export const ProfileIdSchema = z.string().uuid();

// 1. Exportable sync parsing step
export const parseProfileId = (raw: unknown): Result<string, Error> =>
  trySync(() => ProfileIdSchema.parse(raw), toError)();

// 2. Exportable async loading step
export const loadProfileRow =
  (fetchRow: (id: string) => Promise<unknown>) =>
  (ctx: { profileId: string }): ResultAsync<unknown, Error> =>
    tryAsync(fetchRow(ctx.profileId), toError);
```

The workflow plugs them in by name:

```ts title="load-profile.ts"
import { Railway } from "@onrails/result/railway";

const loadProfile = (raw: unknown, fetchRow: (id: string) => Promise<unknown>) =>
  Railway.fromResult("profileId", () => parseProfileId(raw))
    .fromAsync("row", loadProfileRow(fetchRow))
    .done();
```

### Testing Steps Individually
You can test `parseProfileId` and `loadProfileRow` independently by passing direct input/context objects and asserting on the returned `Result`:

```ts title="profile-steps.spec.ts"
import { describe, expect, it } from "bun:test";
import { isErr, isOk } from "@onrails/result";
import { parseProfileId, loadProfileRow } from "./profile-steps";

describe("profile-steps", () => {
  describe("parseProfileId", () => {
    it("successfully parses a valid UUID", () => {
      const validUuid = "123e4567-e89b-12d3-a456-426614174000";
      const result = parseProfileId(validUuid);

      expect(isOk(result)).toBe(true);
      expect(result.value).toBe(validUuid);
    });

    it("returns error on invalid UUID", () => {
      const result = parseProfileId("invalid-id");
      expect(isErr(result)).toBe(true);
    });
  });

  describe("loadProfileRow", () => {
    it("successfully loads a database row", async () => {
      const mockFetch = async (id: string) => ({ id, name: "Ada" });
      const step = loadProfileRow(mockFetch);

      // Pass the context required by this step
      const result = await step({ profileId: "usr_100" }).resolve();

      expect(isOk(result)).toBe(true);
      expect(result.value).toEqual({ id: "usr_100", name: "Ada" });
    });
  });
});
```

## 4. Testing Parallel Execution Steps

Workflows often trigger parallel database queries or API requests. When using `.parallel`, you should test both success merges and failure combinations.

```ts title="parallel-enrichment.spec.ts"
import { describe, expect, it } from "bun:test";
import { Railway } from "@onrails/result/railway";
import { isErr, isOk } from "@onrails/result";

it("merges parallel branch outputs on success", async () => {
  const result = await Railway.context({ userId: "usr_1" })
    .parallel({
      profile: ({ userId }) => Promise.resolve({ id: userId, name: "Alice" }),
      preferences: () => Promise.resolve({ theme: "dark" }),
    })
    .select(({ profile, preferences }) => ({ profile, preferences }));

  expect(isOk(result)).toBe(true);
  expect(result.value).toEqual({
    profile: { id: "usr_1", name: "Alice" },
    preferences: { theme: "dark" },
  });
});

it("bubbles up the first failure in parallel execution", async () => {
  const result = await Railway.context({ userId: "usr_1" })
    .parallel({
      profile: () => Promise.reject(new Error("Database Down")),
      preferences: () => Promise.resolve({ theme: "dark" }),
    })
    .done();

  expect(isErr(result)).toBe(true);
  expect(result.error.message).toBe("Database Down");
});
```

## 5. Dependency Injection & Stubbing I/O (No Mocks Philosophy)

Instead of relying on global monkey-patching libraries (e.g. `jest.mock`), design your workflows around dependency injection. This keeps your workflows pure and makes stubbing boundaries as simple as passing a mock object that returns pre-settled `ResultAsync` objects.

```ts title="user-service.ts"
import { ResultAsync } from "@onrails/result";

export type User = { id: string; email: string };

// The I/O Port
export interface UserRepository {
  getByEmail(email: string): ResultAsync<User, "not_found" | "db_error">;
}

// The Workflow / Service
export const loginWorkflow = (repo: UserRepository, email: string) => {
  return repo.getByEmail(email)
    .map((user) => ({ status: "authenticated" as const, user }));
};
```

You can test this service by passing inlined port implementations:

```ts title="user-service.spec.ts"
import { describe, expect, it } from "bun:test";
import { ResultAsync } from "@onrails/result";
import { loginWorkflow, type UserRepository } from "./user-service";

describe("loginWorkflow", () => {
  it("returns authentication success for valid users", async () => {
    // Port Stub returning pre-settled Ok ResultAsync
    const mockRepo: UserRepository = {
      getByEmail: (email) => ResultAsync.fromOk({ id: "usr_1", email })
    };

    const result = await loginWorkflow(mockRepo, "test@test.com").value;
    expect(result).toEqual({
      _tag: "Ok",
      value: { status: "authenticated", user: { id: "usr_1", email: "test@test.com" } }
    });
  });

  it("bubbles up domain errors safely", async () => {
    // Port Stub returning pre-settled Err ResultAsync
    const mockRepo: UserRepository = {
      getByEmail: () => ResultAsync.fromErr("not_found" as const)
    };

    const result = await loginWorkflow(mockRepo, "test@test.com").value;
    expect(result).toEqual({ _tag: "Err", error: "not_found" });
  });
});
```

## 6. Table-Style Outcome Mapping (`it.each`)

Since ROP maps multiple failure states to a unified error union, testing HTTP or response mappings is clean when using table-style parameterization. Use `it.each` to check the translation of every error track.

```ts title="http-mapping.spec.ts"
import { expect, it } from "bun:test";
import { match } from "@onrails/result";

type DomainError = 
  | { kind: "invalid_input"; field: string }
  | { kind: "unauthorized" }
  | { kind: "not_found" }
  | { kind: "fatal"; reason: string };

const cases = [
  { label: "maps invalid input to 400", err: { kind: "invalid_input" as const, field: "email" }, expected: 400 },
  { label: "maps unauthorized to 401", err: { kind: "unauthorized" as const }, expected: 401 },
  { label: "maps missing resource to 404", err: { kind: "not_found" as const }, expected: 404 },
  { label: "maps fatal errors to 500", err: { kind: "fatal" as const, reason: "disk full" }, expected: 500 },
] as const;

it.each(cases)("$label", ({ err, expected }) => {
  // Translate domain errors to HTTP statuses via exhaustive match
  const status = match(
    // We pass a mock Result containing our test error
    { _tag: "Err" as const, error: err },
    () => 200,
    (e: DomainError) => {
      switch (e.kind) {
        case "invalid_input": return 400;
        case "unauthorized": return 401;
        case "not_found": return 404;
        case "fatal": return 500;
      }
    }
  );
  expect(status).toBe(expected);
});
```

## 7. Property-Based Pipeline Invariants (`fast-check`)

When verifying algebraic laws, normalizations, or round-trip properties across a pipeline, property-based tests are much stronger than individual unit cases.

```ts title="normalization.spec.ts"
import { expect, it } from "bun:test";
import * as fc from "fast-check";
import { ok, err, map, type Result } from "@onrails/result";

// 1. Build an Arbitrary that yields Result values
const arbResult: fc.Arbitrary<Result<number, string>> = fc.oneof(
  fc.integer().map((n) => ok<number, string>(n)),
  fc.string().map((s) => err<number, string>(s))
);

// 2. Assert Functor Identity: map(x => x)(r) === r
it("satisfies the Functor Identity law", () => {
  fc.assert(
    fc.property(arbResult, (r) => {
      const identity = (x: number) => x;
      expect(map(identity)(r)).toEqual(r);
    })
  );
});
```

## 8. Compile-Time Type Inference Verification (`ts-expect`)

TypeScript's type-inference is a core part of the `@onrails` API contract. You should write compile-time assertions to verify these type boundaries.

```ts title="types.spec.ts"
import { expectType, type TypeEqual } from "ts-expect";
import { Railway } from "@onrails/result/railway";
import { type Result } from "@onrails/result";

function _typeChecks() {
  const workflow = Railway
    .fromSync("id", () => "usr_1", (e) => new Error(String(e)))
    .derive("slug", ({ id }) => `${id}-slug`)
    .select(({ id, slug }) => ({ id, slug }));

  // 1. Assert the exact return type structure
  expectType<TypeEqual<typeof workflow, Result<{ id: string; slug: string }, Error>>>(true);

  // 2. Negative type checking: verify that context fields are readonly
  // @ts-expect-error — Context fields are readonly
  workflow.value.id = "new-id";
}
```

<Callout type="warn">
  Make sure your `tsconfig.json` `include` option references your test directory (e.g. `test/**/*.spec.ts`). If test files are excluded, `tsc --noEmit` will silently skip them during type-checking, and broken types may slip to production.
</Callout>

## Testing Strategy Matrix

| Dimension | Tooling | Recommended Target |
| :--- | :--- | :--- |
| **Logic & Short-Circuits** | `bun:test` / Vitest + `isOk`/`isErr` | Workflows with sync/async conditional boundaries. |
| **Isolated Steps** | `bun:test` / Vitest step calls | Reusable step functions plugged into `Railway` workflows. |
| **Exhaustive Mapping** | `it.each` tables | Enumerable branches (translating error tracks to HTTP codes). |
| **Algebraic Invariants** | `fast-check` properties | Idempotent normalizations, round-trips, encoders/decoders. |
| **Type Contracts** | `ts-expect` + `tsc` | Inferred types, error union boundaries, context read-only invariants. |
