# AGENTS.md

Operating rules for human + AI contributors. Repo-local; overrides any external skill defaults.

## What this repo is

`onrails` defines the railway-oriented primitives for Alan's TypeScript stack — `Result` / `ResultAsync`, `Maybe`, exhaustive `match`, plus a codemod and ESLint plugin to migrate consumers off `neverthrow`. Library-only; no runtime IO, no servers, no DB.

Per-package design notes:

- [`@onrails/result`](packages/result/DESIGN.md)
- [`@onrails/maybe`](packages/maybe/DESIGN.md)
- [`@onrails/pattern`](packages/pattern/DESIGN.md)

## Toolchain

- **Bun ≥ 1.0** required. Use `bun`, `bun run`, `bun test`, `bunx`. No npm/yarn/pnpm.
- **Biome** for lint + format ([`biome.json`](biome.json)). Run `bun lint:fix` before pushing.
- **TypeScript strict** — `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `verbatimModuleSyntax` ([`tsconfig.base.json`](tsconfig.base.json)).
- **`bun:test`** for runtime tests; `ts-expect` (`expectType` / `TypeEqual`) for compile-time type tests.
- **Husky** installs hooks via `bun install`. `pre-commit` runs lint+typecheck; `commit-msg` runs commitlint.

Root scripts fan out to packages:

```
bun typecheck   # bun run --filter '@onrails/*' typecheck
bun test        # bun run --filter '@onrails/*' test
bun lint        # biome check .
bun check       # typecheck + lint + test
```

Each package owns its own `typecheck`/`test`/`check` scripts.

## Quality bar

Code-review blockers, not preferences:

- No `any`. Biome enforces `noExplicitAny: error` (off in `*.spec.ts` only).
- No `as unknown as` without an inline comment naming the constraint that makes it safe (e.g. a runtime `_tag` check just above).
- No `@ts-ignore` / `@ts-expect-error` without an inline comment naming the bug or upstream issue.
- No `throw` in business logic. Wrap third-party throws at the boundary with `trySync` / `ResultAsync.fromPromise`.
- No `Promise<Result<…>>` in public signatures — return `ResultAsync<T, E>`. The `@onrails/eslint-plugin` enforces this rule on consumers; the lib itself uses `Promise<Result<…>>` only inside `ResultAsync` internals.
- No new runtime dependencies without discussion. The published surface is small by design.
- No AI attribution, co-author footers, "Generated by" comments, or emoji in commits, PRs, or code.

## Type + error model (the lib's own dogfood)

This repo is the canonical implementation, so the rules it imposes on consumers apply here too:

- `Result<T, E>` / `ResultAsync<T, E>` at every boundary that can fail.
- `Maybe<T>` for expected absence (cache miss, optional relation, no match). Not for failure.
- Tagged unions over classes — `{ readonly _tag: "Ok"; readonly value }` / `{ readonly _tag: "Err"; readonly error }`. `readonly` on everything.
- `flatMap` is canonical; `andThen` (neverthrow muscle memory) and `chain` (Fantasy Land) are aliases — keep all three.
- Module-function default; fluent (`/fluent` subpath) is optional sugar. Curried + uncurried pairs (`map` curried, `mapResult` data-first).
- Subpath exports only — no deep imports into `src/*` from consumers, no relative imports across packages.

## Tests

- **Runtime tests**: `bun:test`. Files named `*.spec.ts` under `packages/<pkg>/test/`.
- **Compile-time type tests**: `ts-expect` (`expectType` + `TypeEqual`). Use `TypeEqual<X, Y>` for exact-equality assertions on public type surface; fall back to plain `expectType<T>(value)` (assignability) when curry/contextual inference makes exact equality brittle. Type tests live in `test/types.spec.ts` per package.
- **All test files must be in `tsconfig.json` `include`**. Each package's `tsconfig.json` includes `["src/**/*.ts", "test/**/*.ts"]` so `tsc --noEmit` covers tests. `bun test` transpiles without enforcing types, so any package that excludes tests from `tsc` will silently ship broken type assertions.
- **Property-based tests** (`fast-check`): the default for algebraic laws, invariants, and round-trip properties — anywhere a hand-picked example is just one point in a space of valid inputs. Functor/monad laws, `combine`/`combineTuple` fold semantics, `fromNullable` ↔ `match` round-trip, exhaustive-match coverage. Build per-package arbitraries (e.g. `arbResult`) and reuse across files.
- **Table-style tests** (`it.each` / `describe.each`): the default for closed enumerable branches — primitive dispatch, format/serialisation rules, status-code mapping, falsy-value handling. One row per case with a `label` field; the test body is a single assertion. Better than N near-identical `it` blocks.
- **Plain `it` blocks**: keep for one-off scenarios, integration cases that don't generalise, and tests that exist to document a specific bug fix.
- **No mocks for the lib's own functions.** Use the real `Result` / `Maybe` / `match` in tests. The library has no IO to stub.
- **`*.spec.ts` opts out of `noExplicitAny`** ([`biome.json`](biome.json)). Use `any` only when narrowing through `fast-check` arbitraries or test-only utilities forces it; production code stays clean.

## PRs

- One package per PR where possible.
- Behavior changes need tests (`bun:test` under `packages/<pkg>/test/`).
- Compat / shim changes extend `packages/result/test/neverthrow-conformance.spec.ts`.
- Open an issue before adding exports or changing the published surface.

## Releases

Automated by [release-please](https://github.com/googleapis/release-please-action). A `feat`/`fix` on `main` opens a per-package release PR; merging publishes via `bun publish` with npm provenance. Do not bump versions manually.
