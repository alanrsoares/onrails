# AGENTS.md

Operating rules for human + AI contributors. Repo-local; overrides external skill defaults.

## What this repo is

`onrails` is the railway-oriented primitive set for TypeScript — `Result` / `ResultAsync`, `Maybe`, exhaustive `match`, plus a codemod, ESLint plugin, and Biome plugin to migrate consumers off `neverthrow`. Library-only; no runtime IO, no servers, no DB.

Per-package design notes: [result](packages/result/DESIGN.md) · [maybe](packages/maybe/DESIGN.md) · [pattern](packages/pattern/DESIGN.md). Worked composition examples: [result/RECIPES.md](packages/result/RECIPES.md).

## Toolchain

- **Bun ≥ 1.0**. Use `bun`, `bun run`, `bun test`, `bunx`. No npm/yarn/pnpm.
- **Biome** for lint + format ([`biome.json`](biome.json)). Root config dogfoods `@onrails/biome-plugin` grit rules (with `result` internals excluded). Run `bun lint:fix` before pushing.
- **TypeScript strict** — `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `verbatimModuleSyntax` ([`tsconfig.base.json`](tsconfig.base.json)).
- **`bun:test`** runtime; **`ts-expect`** (`expectType` / `TypeEqual`) compile-time.
- **Husky** installs hooks on `bun install`: `pre-commit` runs `bun check`, `commit-msg` runs commitlint.

Root scripts fan out:

```
bun typecheck   # bun run --filter '@onrails/*' typecheck
bun test        # bun run --filter '@onrails/*' test
bun lint        # biome check . --error-on-warnings
bun format      # biome format --write .
bun check       # lint + typecheck + test
```

## Quality bar

Code-review blockers, not preferences:

- No `any`. Biome enforces `noExplicitAny: error` (off in `*.spec.ts`).
- No `as unknown as` without an inline comment naming the constraint that makes it safe.
- No `@ts-ignore` / `@ts-expect-error` without an inline comment naming the bug or upstream issue.
- No `throw` in business logic. Wrap third-party throws at boundaries with `trySync` / `tryAsync`.
- No `Promise<Result<…>>` in public signatures — return `ResultAsync<T, E>`. Enforced for consumers by `@onrails/eslint-plugin` and `@onrails/biome-plugin`.
- No new runtime dependencies without discussion.
- No AI attribution, co-author footers, "Generated by" comments, or emoji in commits, PRs, or code. The commit-msg hook strips `Co-authored-by` lines for AI tools. Do not commit `.claude/` or `.cursor/` (gitignored).

## Type + error model

The lib eats its own dogfood:

- `Result<T, E>` / `ResultAsync<T, E>` at every fallible boundary.
- `Maybe<T>` for expected absence (cache miss, optional relation, no match). Not for failure.
- Tagged unions over classes — `{ readonly _tag: "Ok"; readonly value }` / `{ readonly _tag: "Err"; readonly error }`. `readonly` everywhere.
- `flatMap` is canonical. Class methods expose `.andThen` for neverthrow muscle memory.
- Module-function default; `/fluent` subpath is optional sugar. Every transform (`map`, `flatMap`, `match`, …) is **dual-form**: `map(r, fn)` data-first or `map(fn)(r)` curried.
- Subpath exports only — no deep `src/*` imports from consumers, no relative cross-package imports.

### Domain language

Use these terms; avoid the listed synonyms.

- **Railway-compliant building block** — primitive or helper that composes expected failures through `Result` / `ResultAsync` without throwing across the public boundary. _Avoid_: universal error wrapper, exception replacement.
- **Expected failure** — modeled failure callers inspect and handle as normal workflow. _Avoid_: defect, exception, absence.
- **Expected absence** — missing value that is normal and non-diagnostic. _Avoid_: failure, error.
- **Defect** — unexpected thrown exception or rejected promise from a boundary or programmer mistake. Converted into expected failures at explicit boundaries; not part of ordinary domain flow. _Avoid_: expected failure, validation error.

Worked example: parser on malformed input → `Result` with expected failure (caller needs diagnostic info). Cache miss → `Maybe` (expected absence). Third-party throw inside `JSON.parse` → defect until wrapped by `trySync` at the boundary.

```
result | maybe | pattern | codemod | eslint-plugin | biome-plugin | repo | deps | ci | docs
```

Subject ≤ 50 chars. Body explains *why* when non-obvious. No `body-max-line-length` cap.

## Tests

- **Runtime**: `bun:test`. `*.spec.ts` under `packages/<pkg>/test/`.
- **Compile-time**: `ts-expect` (`expectType` + `TypeEqual`). Per-package `test/types.spec.ts`. Every test file must be in `tsconfig.json` `include` — `bun test` transpiles without type-checking, so excluded tests silently ship broken assertions.
- **Property-based** (`fast-check`): default for algebraic laws, invariants, round-trips. Build per-package arbitraries (e.g. `arbResult`) and reuse.
- **Table-style** (`it.each` / `describe.each`): default for closed enumerable branches. One row per case with a `label` field, single-assertion body.
- **Plain `it`**: one-off scenarios, integration cases that don't generalise, regression tests for specific bug fixes.
- **No mocks for the lib's own functions** — the library has no IO to stub.
- **`*.spec.ts` opts out of `noExplicitAny`** ([`biome.json`](biome.json)). Use sparingly; production stays clean.

## PRs

- One package per PR where possible.
- Behavior changes need tests.
- Compat changes extend [`packages/result/test/neverthrow-conformance.spec.ts`](packages/result/test/neverthrow-conformance.spec.ts).
- Open an issue before adding exports or changing the published surface.

## Releases

Automated by [release-please](https://github.com/googleapis/release-please-action). A `feat`/`fix` on `main` opens a per-package release PR; merging publishes via `bun publish` with npm provenance. Do not bump versions manually.

**Squash merges classify on the PR title, not the commits.** release-please reads only the squash commit header (= the PR title), so a PR whose body carries a `feat`/`fix` but is *titled* `chore` ships nothing. Title every PR with the highest releasable type it contains (a `feat(maybe)` inside → title it `feat(maybe): …`), or the release is silently skipped.
